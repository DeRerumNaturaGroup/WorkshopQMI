Background: Graph Neural Networks (GNNs)

  • What are GNNs?
  • ELI5 - A game
  • Our First GNN (Layer)
  • 🟣 Summary

Graph Neural Networks (GNNs) are a class of deep learning models that operate on graph-structured data.

Defining attributes – A GNN is a neural network with two defining attributes:

  • Its input is a graph
  • Its output is permutation equivariant

⭕ Permutation means re-ordering of nodes.

Jargon Alert Permutation Equivariance is a fancy way of saying: If you shuffle the inputs, the output should get shuffled in the same way.

⭕ In the methanol example, we could have made the carbon be atom 1 instead of 4. The adjacency matrix would change.

  • If you are trying to model a per-atom quantity (e.g like partial charge or chemical shift), the output should change too. Atom 4’s properties would now be Atom 1’s.
  • But that shouldn’t change the molecule’s physical properties. This should be invariant to changing the order of the atoms

Q How can we make an equivariant model invariant?

  • (1) Message Passing
  • (2) Update Your State
  • (3) Repeat

To understand GNNs, let’s consider a simple game. We have a group of participants, each with a secret number. The goal is to find out what the group is thinking without ever leaving the seat.

  • Take out a small piece of paper. Without telling anyone, write down any whole number between 0 and 100.

  • Now create a table of your neighbors.

    Neighbour Secret Number
    Alice
    Bob
    Charles

  • At this point:

    • You know your own number.
    • You know who your neighbours are.
    • You do not know your neighbours’ numbers.

  • Now each participant tells only their assigned neighbours their secret number. But do not communicate with anyone else.

  • Record your neighbours’ numbers.

    Neighbour Secret Number
    Alice 12
    Bob 45
    Charles 84

  • Using only your own number and those of your neighbours, answer the following questions.

    • Is your number larger or smaller than the average of your neighbours?
    • Is your neighbourhood generally high-valued or low-valued?
    • What do you predict about the rest of the room?

  • Notice that your prediction is based only on local information.

Compute

The average of your own number and all of your neighbours’ numbers.

  • Replace your original number with this new value.
  • Everyone does this at the same time.

We repeat this process three or four times.

  1. Exchange your updated number with your neighbours.
  2. Compute the average of

  • your own updated number,

  • all of your neighbours’ updated numbers.

  • Observe how the numbers begin to converge. Without realizing it, you have implemented the core operation of a Graph Neural Network (GNN).

Workshop Graph Neural Network
Participant Node
Neighbour Edge
Secret number Node feature
Sharing your number Message passing
Averaging neighbour values Aggregation
New averaged number Updated node embedding
Repeating the process Multiple GNN layers

Remarkably:

No participant ever sees the entire network.

Yet, after several rounds of local communication, every participant’s value contains information originating from increasingly distant parts of the graph.

This is exactly how Graph Neural Networks learn from molecules, crystals, social networks, citation networks, and many other graph-structured datasets.

Instead of people exchanging numbers, computers exchange vectors representing atoms, molecules, or other connected entities.

  • We can now try to seek a mathematical formulation for a simple GNN layer.

\[ \begin{equation} f_k = \sigma\left( \sum_i \sum_j v_{ij}w_{jk} \right) \end{equation} \]

  • This will yield a single feature vector for the graph.
  • Is this equation permutation invariant? Yes, because the node index in our expression is index \(i\) which can be re-ordered without affecting the output.

⭕

Alternative – What if we make a small change…

⭕

\[ f_k = \sigma\left( \sum_i v_{ij}w_{ik} \right) \]

Problem?

  • We have one weight vector per node now.
  • The trainable weights depend on the ordering of the nodes.
  • If we swap the node ordering, our weights will no longer align

If we input two methanol molecules, but switched the labels of two atoms, we would get different answers.

This then is not permutation invariant.

These simple examples differ from real GNNs in two important ways:

  1. They give a single feature vector output, which throws away per-node information,
  2. They do not use the adjacency matrix.

Let’s see a real GNN that has these properties while maintaining permutation invariance — or equivariance (swapping inputs swaps outputs the same way).

Graph Convlution Networks (GCNs)

Graph Convolution Networks

How they work Graph Convolution Code Materials to Model Projects

  • 🟤 Overview
  • 🟣 By Analogy
  • 🟢 Animation

GCNs try to incorporate neighbor information into each node’s representation.

  • Initially the output for node i will depend only on its immediate neighbors.
  • While only immediate neigbors is not enough for chemistry, you can stack multiple layers. If you have two layers, the output for node will include information about node’s neighbors’ neighbors.
  • The averaging process in GCNs accomplishes two goals:
    1. it gives permutation equivariance by removing the effect of neighbor order and
    2. it prevents a change in magnitude in node features. A sum would accomplish (i) but would cause the magnitude of the node features to grow after each layer. Of course, you could ad-hoc put a batch normalization layer after each GCN layer to keep output magnitudes stable but averaging is easy.

Why does stacking layers help?

  • What does a single GCN layer do? It updates each node by: > “Looking at its immediate neighbors and mixing their information.”

  • So after 1 layer:

    • Each node knows about its 1-hop neighbors

  • But we can ‘stack’ GCN layers. What does that mean? Simply that we can apply a GCN layer to the output of another GCN layer. So:

    Layer 1 → node sees 1-hop neighbors Layer 2 → node sees 2-hop neighbors Layer 3 → node sees 3-hop neighbors etc.

  • Thus each layer expands how far information flows:

    • Before any layers: Each node only knows itself.
    • After 1 layer: Each node knows itself + immediate neighbors
    • After 2 layers: Each node knows neighbors of neighbors
    • After k layers: Each node knows: k-hop neighborhood structure

  • Why not just use one big layer? Because:

    1. Local structure matters first
    • chemical bonds
    • social connections
    • protein interactions
    1. Multi-hop structure is important too. Many properties depend on higher-order structure and these appear only after stacking layers.:
    • molecular rings
    • functional groups
    • long-range interactions
    1. Expressivity increases. Each layer adds
    • nonlinear transformation (weights + activation)
    • new feature mixing

    So deeper GCN = richer representations.

Procedure

The procedure for our GCN has two steps.

Each node starts with its own “info”

Every atom starts with a little notebook of numbers (its feature vector). At the start, it might just say:

Hydrogen = [1 0 0]

Oxygen = [0 1 0]

So each atom only knows what it is, not what it’s connected to.

Next each atom gets a “summary of its neighborhood.”

⭕ Every atom goes on a spree to collect its neighbor’s information.

  • collects their feature vectors
  • averages them

Since order doesn’t matter in averaging, the process is permutation equivariant.

Q Why average? Why not just add?

Suppose you took an exam, you find out your score. You want to figure out how you did in relation to your peers.

  • You start with your ‘own information’
  • You go to your neighbors and gather their information (ethically of course)
  • You now have a guess as to how the exam went, but wait…
  • Your friends have been gathering information in their networks too.
  • When they update their information, you realize that you have to update yours too.

But simple averaging is too dumb

Some neighbors are more important. So we add learning:

Before averaging, we multiply each neighbor’s info by a trainable matrix

These weights are what we have to ‘learn’.

In one sentence

A GCN lets each node repeatedly:

mix its own features with an averaged, learnable summary of its neighbors, so it gradually learns its local environment.

Define a new neural-network layer called GCNLayer.

  • Inherits from nn.Module, the base class for all PyTorch models and layers.
  • This allows the layer to store trainable parameters and participate in automatic differentiation.
    def __init__(self, in_features, out_features, activation=None):

Constructor for the layer.

Parameters:

  • in_features: Number of input features per node.
  • out_features: Number of output features per node.
  • activation: Optional activation function such as F.relu.
        super(GCNLayer, self).__init__()

Initialize the parent nn.Module class.

This gives the layer access to PyTorch functionality such as parameter tracking, device management, and automatic differentiation.

        self.w = nn.Parameter(torch.empty(in_features, out_features))

Create the trainable weight matrix.

Shape:

(in_features, out_features)

Each node feature vector will eventually be multiplied by this matrix to produce a new feature representation.

Because it is wrapped with nn.Parameter, PyTorch automatically treats it as a learnable parameter.

        nn.init.xavier_uniform_(self.w)

Initialize the weight matrix using Xavier (Glorot) initialization.

This chooses small random values with a variance designed to keep activations from becoming too large or too small during training.

        self.activation = activation

Store the activation function.

Examples:

  • F.relu
  • torch.sigmoid
  • torch.tanh

If activation=None, no activation will be applied.

    def forward(self, nodes, adj):

Define how data flows through the layer.

Inputs:

  • nodes: Node feature matrix.
  • adj: Adjacency matrix describing graph connectivity.

This method is automatically called when the layer is used.

        degree = torch.sum(adj, dim=-1)

Compute the degree of each node.

The degree is the number of neighbors connected to a node.

For a batched adjacency matrix:

(batch_size, num_nodes, num_nodes)

the result has shape:

(batch_size, num_nodes)
        new_nodes = torch.einsum(
            "bi,bij,bjk,kl->bil",
            1 / degree,
            adj,
            nodes,
            self.w
        )

Perform the graph convolution.

This combines four operations:

  1. Normalize by node degree (1/degree).
  2. Gather information from neighboring nodes using adj.
  3. Combine neighbor features using nodes.
  4. Transform features using the learnable weight matrix self.w.

The Einstein summation notation:

"bi,bij,bjk,kl->bil"

specifies how the tensors should be multiplied and summed.

Output shape:

(batch_size, num_nodes, out_features)

This line is the core of the GCN layer.

        if self.activation is not None:

Check whether an activation function was supplied.

If no activation function was provided, this block is skipped.

            new_nodes = self.activation(new_nodes)

Apply the activation function element-wise to every output feature.

For example, if activation=F.relu:

ReLU(x) = max(0, x)

Negative values become zero while positive values remain unchanged.

        return new_nodes, adj

Return the updated graph representation.

  • new_nodes: New node features after graph convolution.
  • adj: Original adjacency matrix.

Returning adj unchanged makes it easy to stack multiple GCN layers together.

  • Some of the code above PyTorch boilerplate. The key lines are the ones highlighted.

    degree = torch.sum(adj, dim=-1)

    computes the degree by summing over the columns of the adjacency matrix

    new_nodes = torch.einsum("bi,bij,bjk,kl->bil", 1 / degree, adj, nodes, self.w)

    is the GCN equation

We’ll now revisit predicting solubility with GCNs. Remember before that we used the features included with the dataset. Now we can use the molecular structures directly.

Our GCN layer outputs node-level features.

To predict solubility, we need to get a graph-level feature.

For now let’s just take the average over all node features after our GCN layers. This is simple, permutation invariant, and gets us from node-level to graph level.

class GRLayer(nn.Module):
    """A GNN layer that computes average over all node features"""

    def __init__(self):
        super(GRLayer, self).__init__()

    def forward(self, nodes, adj):
        reduction = torch.mean(nodes, dim=1)
        return reduction

The key line in that code is to just to compute the mean over the nodes (dim=1):

reduction = torch.mean(nodes, dim=1)

To complete our deep solubility predictor, we can add some dense layers and make sure we have a single-output without activation since we’re doing regression. We’ll define a PyTorch module that combines the GCN layers, graph readout, and dense layers.

class SolubilityModel(nn.Module):
    def __init__(self, node_features=100):
        super(SolubilityModel, self).__init__()
        self.gcn1 = GCNLayer(node_features, node_features, activation=F.relu)
        self.gcn2 = GCNLayer(node_features, node_features, activation=F.relu)
        self.gcn3 = GCNLayer(node_features, node_features, activation=F.relu)
        self.gcn4 = GCNLayer(node_features, node_features, activation=F.relu)
        self.gr = GRLayer()
        self.fc1 = nn.Linear(node_features, 16)
        self.fc2 = nn.Linear(16, 1)
    
    def forward(self, nodes, adj):
        x, adj = self.gcn1(nodes, adj)
        x, adj = self.gcn2(x, adj)
        x, adj = self.gcn3(x, adj)
        x, adj = self.gcn4(x, adj)
        x = self.gr(x, adj)
        x = torch.relu(self.fc1(x))
        x = self.fc2(x)
        return x

model = SolubilityModel()

Where does the 100 come from? Well, this dataset has lots of elements so we cannot use our size 3 one-hot encodings because we’ll have more than 3 unique elements. We previously only had C, H and O.

This is a good time to update our smiles2graph function to deal with this.

def gen_smiles2graph(sml):
    """Argument for the RD2NX function should be a valid SMILES sequence
    returns: the graph
    """
    m = rdkit.Chem.MolFromSmiles(sml)
    m = rdkit.Chem.AddHs(m)
    order_string = {
        rdkit.Chem.rdchem.BondType.SINGLE: 1,
        rdkit.Chem.rdchem.BondType.DOUBLE: 2,
        rdkit.Chem.rdchem.BondType.TRIPLE: 3,
        rdkit.Chem.rdchem.BondType.AROMATIC: 4,
    }
    N = len(list(m.GetAtoms()))
    nodes = np.zeros((N, 100))
    for i in m.GetAtoms():
        nodes[i.GetIdx(), i.GetAtomicNum()] = 1

    adj = np.zeros((N, N))
    for j in m.GetBonds():
        u = min(j.GetBeginAtomIdx(), j.GetEndAtomIdx())
        v = max(j.GetBeginAtomIdx(), j.GetEndAtomIdx())
        order = j.GetBondType()
        if order in order_string:
            order = order_string[order]
        else:
            raise Warning("Ignoring bond order" + order)
        adj[u, v] = 1
        adj[v, u] = 1
    adj += np.eye(N)
    return nodes, adj
nodes, adj = gen_smiles2graph("CO")
nodes_t = torch.from_numpy(nodes[np.newaxis, ...]).float()
adj_mat_t = torch.from_numpy(adj[np.newaxis, ...]).float()
model(nodes_t, adj_mat_t)

We use a PyTorch Dataset class, which requires implementing len and getitem methods:

from torch.utils.data import Dataset, DataLoader

class SolubilityDataset(Dataset):
    def __init__(self, data, start_idx=0, end_idx=None):
        self.data = data
        self.start_idx = start_idx
        self.end_idx = end_idx if end_idx is not None else len(data)
    
    def __len__(self):
        return self.end_idx - self.start_idx
    
    def __getitem__(self, idx):
        idx = idx + self.start_idx
        nodes, adj = gen_smiles2graph(self.data.SMILES[idx])
        sol = self.data.Solubility[idx]
        # this time we do not batch - it will be done by dataloader
        nodes_t = torch.from_numpy(nodes).float()
        adj_t = torch.from_numpy(adj).float()
        return (nodes_t, adj_t), torch.tensor(sol, dtype=torch.float32)

Now we can split our dataset into train, validation, and test sets. I just used 200/200 for testing and validation here.

test_data = SolubilityDataset(soldata, 0, 200)
val_data = SolubilityDataset(soldata, 200, 400)
train_data = SolubilityDataset(soldata, 400)
train_loader = DataLoader(train_data, batch_size=1, shuffle=True)
val_loader = DataLoader(val_data, batch_size=1, shuffle=False)

optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
criterion = nn.MSELoss()

history = {"loss": [], "val_loss": []}
epochs = 10

for epoch in range(epochs):
    model.train()
    train_loss = 0
    for (nodes, adj), y in train_loader:
        optimizer.zero_grad()
        y_pred = model(nodes, adj)
        loss = criterion(y_pred.squeeze(), y)
        loss.backward()
        optimizer.step()
        train_loss += loss.item()
    
    model.eval()
    val_loss = 0
    with torch.no_grad():
        for (nodes, adj), y in val_loader:
            y_pred = model(nodes, adj)
            loss = criterion(y_pred.squeeze(), y)
            val_loss += loss.item()
    
    train_loss /= len(train_loader)
    val_loss /= len(val_loader)
    history["loss"].append(train_loss)
    history["val_loss"].append(val_loss)
    print(f"Epoch {epoch+1}/{epochs}, Train Loss: {train_loss:.4f}, Val Loss: {val_loss:.4f}")

Question: Is the data overfit or underfit?

This model is definitely underfit. One reason is that our batch size is 1. This is a side-effect of making the number of atoms variable, because otherwise we have trouble batching together our data if there are two unknown dimensions. A standard trick is to group together multiple molecules into one graph, but making sure they are disconnected (no bonds between the molecules). That allows you to batch molecules without increasing the rank of your model/data.
